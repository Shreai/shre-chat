# Architecture Decisions

This document records major technical decisions so future teams know why the system looks the way it does.

## Purpose

- capture important design choices
- reduce repeated debates
- keep platform history understandable
- make tradeoffs explicit

## Decision Record Fields

- decision id
- date
- title
- context
- decision
- alternatives considered
- consequences
- owner
- review date

## Rules

- log major decisions before or during implementation
- keep the record short and concrete
- note rejected alternatives when the choice matters
- update the record if the decision changes materially

## Good Decision Topics

- database choice
- API style
- auth model
- deployment target
- tenant model
- vector storage choice
- agent routing choice
- caching or compression strategy

## Rule

If a technical choice is important enough to affect future work, write it down once.
